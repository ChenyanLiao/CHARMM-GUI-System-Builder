## Implemented scope

- Added a risk-ranked decision inventory covering PDB Reader, Ligand Reader, Membrane Builder, Solution Builder, Quick Bilayer, and GROMACS output controls.
- Added reusable target profiles and immutable, hash-locked build contracts with input provenance, decision records, approval state, and explicit production blockers.
- Added scoped, expiring execution authorization backed by an OS-vault signing-key reference. Editable state fields cannot authorize side effects.
- Added a runtime capability handshake so Codex, Claude, OpenClaw, Hermes, and generic agents can fail closed when browser, API, vault, download, or durable-wait capabilities are unavailable.
- Added a documented official-API route for login, Quick Bilayer submission, job status, and download. Full PDB Reader, Ligand Reader, Membrane Builder, and Solution Builder remain audited browser-assisted routes because their complete official submission APIs are not documented in the project registry.
- Added one-submission protection. A timeout or response without a job ID records `submission_uncertain` and blocks blind retries.
- Bound status checks and downloads to the same locked contract, signed authorization, and persisted job ID.
- Added atomic state writes and streamed downloads through a `.partial` file before atomic rename.
- Added non-destructive v1 state migration with secret redaction and JSON plus Markdown migration reports.
- Strengthened final-package gates: a candidate package needs archive and component validation; technical completion additionally requires strict `grompp`, plus custom-ligand verification when expected.
- Preserved the scientific boundary: package generation and technical validation cannot automatically approve production or establish a binding site.

## Installation and rollback proposal

Installation was not performed. Before a separately approved installation:

1. Back up the installed `1.1.1` directory to a timestamped sibling directory.
2. Copy the repository contents into a new temporary lowercase directory.
3. Run strict package validation in that directory.
4. Atomically replace the installed directory only after validation passes.
5. Retain the backup until v2.1.0 has passed a local dry-run and test-only acceptance check.

Rollback is an atomic restoration of the timestamped `1.1.1` backup. No compatibility copy should be left in the repository or Documents directory.
